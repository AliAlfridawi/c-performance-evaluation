# Abstract

This repository benchmarks quicksort, linear search, and binary search across C, Java, and Python using deterministic shared inputs and an instrumented comparison counter. The final review of the harness found that the methodology is defensible for a narrow question: how do three implementations behave when the timed region is limited to the in-process algorithm section and setup costs are intentionally excluded? The current version is materially stronger than the original repository state because it uses shared on-disk datasets, explicit search scenarios, warm-up and measured trials, batching for very small workloads, preserved raw trial data, and strict validation of cross-language consistency.

The regenerated results show that comparison counts match across all three implementations for every published configuration. At `N=5000`, quicksort on random input has median in-process times of about `0.000098 s` in C, `0.000152 s` in Java, and `0.012475 s` in Python under the reviewed Windows-based harness. Linear search scales with hit position, while binary search remains near logarithmic comparison count across cases. These findings support a bounded claim: when algorithmic work is held constant and only the algorithm section is timed, the observed runtime cost still differs substantially across the three environments on this machine.

The benchmark does not support broader claims about end-to-end application latency, universal language rankings, or steady-state JVM behavior. Comparison counts establish workload parity for the algorithms under test, but they do not equalize runtime services such as JIT compilation, garbage collection, interpreter dispatch, cache behavior, or process startup. The documentation therefore presents the experiment as a transparent, reproducible benchmark harness with explicit scope boundaries rather than as a definitive cross-language performance ranking.
