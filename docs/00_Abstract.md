# Abstract

This repository studies a narrow cross-language benchmarking question: when C, Java, and Python execute the same algorithmic work on shared deterministic inputs, what timing differences remain if the timed region is restricted to the in-process algorithm section? The benchmark covers quicksort, linear search, and binary search, and it aligns workload across languages with shared input files plus cross-language comparison-count parity checks.

The current artifact records `5` warm-up trials and `20` measured trials per configuration, batches very short operations until at least `50 ms` of timed work has accumulated, and publishes both raw trial rows and summarized results. The summary layer reports medians together with quartiles, IQR, min/max spread, standard deviation, and a `95%` bootstrap confidence interval for the median. A machine-readable metadata sidecar captures the host, toolchain, build configuration, command templates, and artifact hashes used for the checked-in snapshot.

The regenerated results support three bounded conclusions. First, comparison counts match across C, Java, and Python for every published configuration, which is strong evidence that the algorithmic workload is aligned. Second, substantial timing gaps remain after that workload alignment: at `N=5000` on random quicksort input, median in-process times are about `0.0000983 s` in C, `0.0001523 s` in Java, and `0.0124753 s` in Python. Third, search-case selection materially changes the observed cost profile, so representative reporting must separate `first_hit`, `middle_hit`, `last_hit`, and `miss`.

The study does not claim end-to-end application latency, universal language rankings, or proof of steady-state JVM behavior. It should instead be read as a transparent, reproducible benchmark harness with explicit scope boundaries and stronger internal workload control than a casual cross-language comparison.
